## Motivation

Kicks off Phase 4 of the v1 roadmap: implement the workflow
orchestration engine. Three entity kinds (`WorkflowTemplate`,
`WorkflowInstance`, `StepRecord`), a five-state lifecycle state
machine, a nine-step execution sequence, the `StepExecutor` and
`ConfigLowerer` trait surfaces, and `SubjectContext` threading
from API layer → engine → step record → script argument.

Upstream crates are all published:
- `philharmonic-types 0.3.4` (cornerstone).
- `philharmonic-store 0.1.0` (substrate traits).
- `philharmonic-store-sqlx-mysql 0.1.0` (backend, dev-dep).
- `philharmonic-policy 0.1.0` (provides `Tenant` + `MintingAuthority` markers).
- `philharmonic-connector-common 0.1.0` (landed earlier today;
  the workflow crate does NOT depend on it — the coupling is
  via the `ConfigLowerer` trait, which is defined here. Any
  concrete lowerer the API layer plugs in will bridge both.)

Phase 4 is the single biggest pre-Golden-Week task. Budget: two
to three days of calendar time. Target: published
`philharmonic-workflow 0.1.0`.

## Scope

### In scope (Phase 4)

All seven tasks from ROADMAP §Phase 4:

1. **Three entity kinds** — `WorkflowTemplate`,
   `WorkflowInstance`, `StepRecord`. Exact slot declarations
   from doc 07 §"Three entity kinds":
   - `WorkflowTemplate`: content slots `script`, `config`;
     entity slot `tenant` (pinned); scalar `is_retired: bool,
     indexed`.
   - `WorkflowInstance`: content slots `context`, `args`;
     entity slots `template` (pinned), `tenant` (pinned);
     scalar `status: i64, indexed`.
   - `StepRecord`: content slots `input`, `output`, `error`,
     `subject`; entity slot `instance` (pinned); scalars
     `step_seq: i64, indexed`, `outcome: i64, indexed`.
2. **`SubjectContext`** — exactly as doc 07 defines. `kind:
   SubjectKind`, `id: String`, `tenant_id: EntityId<Tenant>`,
   `authority_id: Option<EntityId<MintingAuthority>>`,
   `claims: JsonValue`. The `Tenant` / `MintingAuthority` type
   parameters come from `philharmonic-policy`; re-use them via
   `philharmonic_policy::{Tenant, MintingAuthority}` (NOT
   re-defining markers here).
3. **`StepExecutor` trait** — `async fn execute(&self, script:
   &str, arg: &JsonValue, config: &JsonValue) ->
   Result<JsonValue, StepExecutionError>`. Two error variants
   (`Transport(String)` — retryable-inconclusive; `ScriptError(String)`
   — recorded-failure). Use `#[async_trait::async_trait]` per
   doc 07.
4. **`ConfigLowerer` trait** — `async fn lower(&self,
   abstract_config: &JsonValue, instance_id:
   EntityId<WorkflowInstance>, step_seq: u64, subject:
   &SubjectContext) -> Result<JsonValue, ConfigLoweringError>`.
   `#[async_trait::async_trait]`.
5. **`WorkflowEngine<S, E, L>`** — three type parameters: store
   (`S: ContentStore + IdentityStore + EntityStore`), executor
   (`E: StepExecutor`), lowerer (`L: ConfigLowerer`). Four
   mutating methods: `create_instance`, `execute_step`,
   `complete`, `cancel`. All async, all take
   `SubjectContext`.
6. **Status transitions** — the five-state machine from doc 07
   §"Status enum" / §"Status transitions". Encode as:
   ```rust
   #[repr(i64)]
   pub enum InstanceStatus {
       Pending = 0,
       Running = 1,
       Completed = 2,
       Failed = 3,
       Cancelled = 4,
   }
   impl InstanceStatus {
       pub fn is_terminal(&self) -> bool { ... }
       fn can_transition_to(&self, next: Self) -> bool { ... }
   }
   ```
   Terminal operations return `WorkflowError::InstanceTerminal`.
7. **Nine-step execution sequence** (doc 07 §"Execution
   sequence"). Critical invariants that require explicit
   tests:
   - **Step record is created BEFORE the instance revision**
     so the step record can pin to the pre-step instance
     revision (doc 07 step 9). Get this ordering wrong and
     pinning breaks.
   - **Malformed result** (missing `context` OR `output` keys
     in the executor's JSON response) is treated as a script
     error (recorded step failure + transition to `Failed`),
     NOT a transport error (no records, caller may retry).
   - **`done: true`** in the script return transitions the
     instance to `Completed`.
8. **Subject-content audit discipline (privacy-by-default).**
   Per ROADMAP §Phase 4 task 7 and doc 07 §"Step record" /
   §"Subject context":
   - The `subject` content slot of `StepRecord` records
     **identifier and authority only**: `kind`, `id`, and
     `authority_id` (plus the `tenant_id` that's implicit in
     the instance scope).
   - **NEVER** persist `claims` into the step record subject
     content. Not configurable, no per-tenant knob.
   - Write an explicit unit test: populate `SubjectContext.claims`
     with distinctive JSON, run `execute_step`, read the step
     record's `subject` content back, assert the `claims`
     field is absent (or empty — pick one shape and test it).

### Out of scope (explicitly)

- **Retry policy** — caller decides (doc 07 §"What the workflow
  crate doesn't do").
- **Authorization** — API layer's job; the engine trusts the
  `SubjectContext` it receives.
- **Scheduling** — external infrastructure.
- **Multi-instance coordination** — scripts handle via
  connectors.
- **Actual JS execution** — the `StepExecutor` trait is the
  hand-off; a concrete implementation using `mechanics-core`
  lives in the API-layer binary, not here.
- **Concrete `ConfigLowerer` impl** — ditto; that's
  `philharmonic-connector-client` in Phase 5.
- **`mechanics-core` dep** — do not add. The workflow crate is
  transport-naive.
- **Connector-common dep** — do not add. The trait contracts
  bridge both; the crate itself stays decoupled.

### `KIND` UUID generation

Every stable `KIND: Uuid` constant in this workspace is generated
via `./scripts/xtask.sh gen-uuid -- --v4`. NOT `uuidgen`, NOT
`python3 -c "import uuid"`, NOT an online generator. One
canonical source means randomness is uniform across sessions and
there's no "oops I committed a wrong one." Three entity kinds →
three `xtask.sh gen-uuid -- --v4` invocations → three KIND
constants. Put them in `entities.rs` with short comments stating
what each identifies and the generation date.

## Tests required

Two tiers, mirroring Phase 2:

### Tier 1 — Mock substrate (fast, always-on)

An in-crate `MockStore` implementing `ContentStore +
IdentityStore + EntityStore` with in-memory backing. Under
`tests/common/` so both tier-1 test files can reuse.
`MockExecutor` + `MockLowerer` implementing the two new traits
with queue-driven responses (push expected responses;
`execute()` / `lower()` pop them). Every branch of the engine
exercised here:

- **Status transitions** — every one of the 7 transitions from
  doc 07 §"Status transitions", both the happy cases and the
  rejected ones (terminal-state-immutable: attempt `execute_step`
  on `Completed`, `Failed`, `Cancelled` → `InstanceTerminal`).
- **Execution sequence ordering** — assert the step record's
  entity-slot `instance` pins to the pre-step instance
  revision, not the post-step one.
- **Subject context propagation** — create an instance with a
  principal subject; execute_step with an ephemeral subject;
  read the step record's `subject` content; assert the
  recorded subject matches the ephemeral subject passed to
  `execute_step`, not the instance creator's subject.
- **Subject-content audit discipline** — the explicit "claims
  never persisted" test described in task 8 above.
- **Malformed-result handling** — mock executor returns
  `{"output": "x"}` (no `context`). Assert engine transitions
  instance to `Failed`, creates a failed step record, returns
  something that indicates script-error category (not
  transport).
- **Transport failure** — mock executor returns `Err(Transport(...))`.
  Assert engine does NOT create any step record or instance
  revision, returns something the caller can distinguish from
  script-error and retry on.
- **`done: true` flow** — mock executor returns `{"context":
  ..., "output": ..., "done": true}`. Assert instance
  transitions to `Completed` after the step record lands.
- **`complete()` and `cancel()`** — Pending and Running
  instances transition correctly; terminal instances reject.

### Tier 2 — Real MySQL substrate (testcontainers, `#[ignore]`)

Under `tests/engine_mysql.rs`. All tests `#[ignore]`'d so
workspace-level `cargo test` skips them; `./scripts/rust-test.sh
--ignored philharmonic-workflow` runs them. Exercise:

- End-to-end `create_instance` → N `execute_step` → final state
  transition, verifying the instance's revision log and the
  step records persist correctly across the real append-only
  MySQL backend.
- Pinning semantics: created step records reference the
  correct instance revision even after further revisions
  accumulate.
- Terminal-state rejection at the real-store layer (ensures the
  logic isn't relying on mock-store quirks).

At minimum 2-3 tier-2 tests. The goal is to verify the engine
doesn't have a mock-substrate-only assumption; it doesn't need
to duplicate tier-1 coverage exhaustively.

## Workspace conventions (authoritative: `docs/design/13-conventions.md`)

- **Edition 2024, MSRV 1.88.** `Cargo.toml` already has these.
- **License `Apache-2.0 OR MPL-2.0`.** Both `LICENSE-*` files
  exist at the crate root.
- **`thiserror`** for library error enums. No `anyhow`.
- **No panics in library code.** This is THE single biggest
  lint-like discipline in the workspace. Every production
  `src/**/*.rs` path:
  - No `.unwrap()` / `.expect()` on `Result`/`Option`. Propagate
    with `?` via a typed `WorkflowError` variant, or
    `.ok_or_else(...)` / `.map_err(...)`.
  - No `panic!` / `unreachable!` / `todo!` / `unimplemented!`
    on reachable paths. Model unreachability at the type
    level (sealed enums, `NonZero<T>`, etc.).
  - No unbounded indexing. Use `.get(...)` → `Option`,
    propagate.
  - No unchecked integer arithmetic. Use `checked_*` /
    `saturating_*` / `wrapping_*` to declare intent. Expect
    `step_seq: u64` to need `.checked_add(1)` somewhere —
    that's correct and intentional.
  - No lossy `as` casts on untrusted widths. `TryFrom` and
    propagate.
  - Narrow, justified exceptions require inline comments.
    (Unlikely any exception is legitimate in Phase 4.)
  Tests / `#[cfg(test)]` / `tests/**.rs` can `.unwrap()`
  freely — panics in tests are the failure-signal mechanism.
  This includes `tests/common/`.
- **Re-export discipline.** Re-export types from direct deps
  that appear in the crate's own public API. `EntityId<T>`,
  `CanonicalJson`, `JsonValue` almost certainly appear and
  should be re-exported from the cornerstone. `Tenant` and
  `MintingAuthority` from `philharmonic-policy` similarly.
  Don't re-export transitive deps.
- **Rustdoc coverage.** Every public item gets a doc comment.
  The cornerstone is at 99%+; match the discipline.
- **POSIX-ish host.** Your runtime is there; noting for
  completeness.
- **Use `./scripts/*.sh` wrappers — no raw `cargo`.** Before
  concluding, run:

  ```sh
  ./scripts/pre-landing.sh philharmonic-workflow
  ```

  Pre-landing runs `fmt --check` + `check` + `clippy -D
  warnings` + workspace test + the per-modified-crate
  `--ignored` phase. The `--ignored` phase exercises the
  testcontainers tier; Docker/Podman must be reachable on the
  host for the MySQL container to come up. If tier-2 tests fail
  due to container startup rather than engine logic, say so in
  your final summary.

## Crypto-sensitivity

Phase 4 is **NOT a Gate-1 crypto-review trigger** — no
primitives, no signing, no encryption, no hashing of
security-sensitive material. The engine is transport-naive and
lowerer-naive.

The one crypto-adjacent invariant is the **subject-content
audit discipline** (task 8 above): NEVER persist `claims` into
step record subject content. This is privacy-by-default, not
crypto-construction, but it's the single irrecoverable-if-wrong
invariant in Phase 4. Get the test for it right, and the
discipline is load-bearing for Phase 8 (`philharmonic-api`).

If implementation nudges into territory that feels
crypto-adjacent (signing something, encrypting something, MAC
over anything), STOP and flag in your final summary — that's a
spec drift, not Phase 4 scope.

## When in doubt

- **ROADMAP §Phase 4 vs. doc 07:** doc 07 wins as the exhaustive
  spec; ROADMAP is a summary.
- **Doc 07 vs. doc 09 on tenant markers:** re-use
  `philharmonic_policy::Tenant` / `MintingAuthority`. Don't
  redefine them.
- **Ambiguity on a scalar type (i64 vs u64):** doc 07 pins
  `status: i64, indexed` and `step_seq: i64, indexed`,
  `outcome: i64, indexed`. The ROADMAP task 6 mentions
  `#[repr(i64)]` enum, and `step_seq` at the API level
  surfaces as `u64` (see the `ConfigLowerer::lower`
  signature). Match doc 07 for on-substrate storage; use
  typed conversions (`TryFrom`) at the API boundary.
  Storage → public: `i64` → `u64` with `TryFrom`. Public →
  storage: `u64` → `i64` with `TryFrom` + propagate error.
  Avoid `as` casts.
- **`CanonicalJson` vs `JsonValue`:** the substrate's content
  slots are canonical-JSON-encoded bytes (JCS per RFC 8785).
  `args` and `context` content slots are `CanonicalJson`
  serialized. The executor's `arg` parameter is `JsonValue`
  (assembled in-memory, never stored as bytes as-is). Follow
  doc 07 and the existing conventions in `philharmonic-policy`
  for the right types at the right layers.
- **Engine dispatch async-trait objects:** async-trait gives
  `dyn`-compat but also means the trait isn't object-safe for
  `Box<dyn Trait>` without `async-trait`'s box-future trick.
  Keep the engine generic on three type parameters; don't
  erase to `dyn`.
