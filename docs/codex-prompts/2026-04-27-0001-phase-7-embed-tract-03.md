# Phase 7 Tier 1 — `embed` rewrite to tract / tokenizers (round 03 dispatch)

**Date:** 2026-04-27
**Slug:** `phase-7-embed-tract`
**Round:** 03 (resume after round-02 stopped at Phase 1 gate;
  extends the public API + build.rs to handle ONNX external-data so bge-m3 stays as the default)
**Subagent:** `codex:codex-rescue`

## Why this round exists

Round 02 stopped at the Phase 1 op-coverage gate per
instructions. The probe found that `BAAI/bge-m3`'s ONNX
export is **not a single self-contained byte blob** — it
ships as `onnx/model.onnx` (~724KB metadata) plus
`onnx/model.onnx_data` (~2.27GB external-weights). The
round-02 architecture assumed a single `include_bytes!` per
model, so tract's `model_for_read(&mut &bytes[..])` (which
has no path context to resolve external-data references)
rejected the load *before* op-coverage was checked.

Yuka's call on 2026-04-27 14:37 JST: **keep bge-m3 as the
default; extend the public API to accept external-data
bytes**. The "merge external data into a single ONNX at
build time" alternative was considered and rejected for
bge-m3 specifically — ONNX's protobuf serialization is
capped at ~2GB per file (32-bit length prefix), which is
*why* bge-m3 ships with external data in the first place.
A merge-at-build-time enhancement may still land later for
sub-2GB external-data models, but it's out of scope here.

Reason: bge-m3 is multilingually decisive for the
deployment target; downgrading to a single-file 470MB
multilingual loses real quality.

## Decisions fixed upstream — supplement to round-02

These extend (not replace) round-02's "Decisions fixed
upstream":

7. **(B-revised) bge-m3 stays as default; public API
   extended for ONNX external-data.** The
   `Embed::new_from_bytes` constructor gains an
   `onnx_external_data: Option<&[u8]>` parameter, located

   build-time merge of external-data into a single ONNX
   for sub-2GB models is a worthwhile generic
   enhancement but explicitly NOT in this round. bge-m3
   doesn't fit a merged ONNX (>2GB protobuf cap), so it
   wouldn't help for the binding default anyway.

## Public surface — revised

Replaces the §"Public surface" code block in round-02's
§"Default-bundled model architecture":

```rust
pub struct Embed { /* private */ }

impl Embed {
    /// Construct from caller-supplied bytes. Always available.
    /// Pass `onnx_external_data: Some(&bytes)` for ONNX models
    /// that ship with an `*.onnx_data` external-weights file
    /// (e.g., models > 2GB like `BAAI/bge-m3`). Pass `None` for
    /// single-file ONNX models.
    pub fn new_from_bytes(
        model_id: impl Into<String>,
        onnx_bytes: Vec<u8>,
        onnx_external_data: Option<&[u8]>,
        tokenizer_json_bytes: &[u8],
        dimensions: usize,
        max_seq_length: usize,
    ) -> Result<Self, ImplementationError>;

    /// Construct from the build-time-bundled default model.
    /// Available only when the `bundled-default-model` feature
    /// is on AND build.rs successfully bundled. The default
    /// model id, dimensions, max_seq_length, and external-data
    /// presence all come from the cached bundle's
    /// `manifest.json` + `config.json`.
    #[cfg(all(feature = "bundled-default-model", embed_default_bundle))]
    pub fn new_default() -> Result<Self, ImplementationError>;

    pub fn model_id(&self) -> &str;
    pub fn dimensions(&self) -> usize;
}
```

## Cache layout — revised

The bundle directory gains an optional `model.onnx_data`
file:

```
$cache_root/
└── <sanitized-model>__<revision-sha-prefix-12>/
    ├── manifest.json     (model id, full revision, fetched-at, sha256 per file)
    ├── model.onnx
    ├── model.onnx_data   (present iff model uses external data)
    ├── tokenizer.json
    └── config.json
```

`manifest.json` records SHA256 for every file actually
present (so the data file SHA is recorded only when it
exists). Cache-hit verification: every listed file must
exist and match its manifest SHA.

For the bge-m3 bundle specifically, **use the root
`tokenizer.json`** (17,098,108 B) — it's the canonical
sentence-transformers tokenizer. Skip the
`onnx/tokenizer.json` variant (17,082,821 B); it differs
slightly and is not what the model card recommends.

## build.rs HTTP stack — revised

Same dep set as round-02 (`ureq` + rustls + `serde_json`
+ `sha2` + `dirs`). Two new behaviors:

1. **Fetch `model.onnx_data` when present.** The HF repo
   may or may not have an `onnx/model.onnx_data` file; the
   ONNX itself is the source of truth. After fetching
   `onnx/model.onnx`, parse its protobuf header (using
   `prost` or by walking the raw bytes) to detect if any
   `TensorProto.data_location == EXTERNAL`. If yes: fetch
   the referenced external file(s) (typically just one,
   named `model.onnx_data`) from HF. If no: skip the data
   fetch.

   Cheaper alternative: HEAD-request
   `https://huggingface.co/<model>/resolve/<revision>/onnx/model.onnx_data`;
   on 200 fetch, on 404 skip. Less robust but simpler.
   Either approach is fine; pick one and document.

2. **Emit additional cargo directives** when external data
   is present:
   - `cargo:rustc-cfg=embed_default_external_data` —
     additive cfg, the lib.rs `new_default` body
     conditionally `include_bytes!`s the data file under
     this cfg.
   - `cargo:rerun-if-changed=$bundle/model.onnx_data` —
     so cache-bumps re-trigger compile.

   When external data is NOT present (e.g., the small
   test model override), neither directive is emitted.
   The lib.rs `new_default` body passes `None` for
   `onnx_external_data` in that case via `#[cfg(not(...))]`.

## Phase 0 supplements

Add to round-02's Phase 0:

- **(0.5) Tract external-data API research.** Before any
  fetching, investigate `tract-onnx 0.22.x`'s API for
  loading a model whose external-data bytes are supplied
  in memory (rather than discoverable via filesystem
  paths). Look at:
  - `tract_onnx::Onnx` / `tract_onnx::onnx()` builder
    methods — does it have a `with_external_data_handler`,
    `with_data_resolver`, or similar callback?
  - `tract_onnx::tract_hir::tract_core::model::Model`
    construction from a pre-built `ModelProto` — is there
    a `model_for_proto_model`-style entrypoint that
    accepts an already-resolved proto?
  - The `tract_onnx::pb::ModelProto` type (if exposed) —
    can we build one ourselves and feed it in?

  Pick the simplest viable API. **Document the chosen
  API path in residual risks** with the exact function
  name + module path so it's reproducible.

  **Fallback if no clean bytes-API exists**: parse
  `ModelProto` via `prost`, walk every
  `TensorProto.data_location == EXTERNAL`, replace the
  `external_data` field with `raw_data` from the supplied
  bytes (using offset/length from the external_data
  metadata). Pass the resulting in-memory ModelProto to
  tract via whatever lower-level API accepts it. Each
  TensorProto's `raw_data` bytes field is bounded by
  ~2GB (protobuf bytes-field cap), which holds for every
  individual tensor in any practical model — bge-m3's
  largest individual tensor is well under that. The
  total in-memory model size CAN exceed 2GB without
  running into the cap because we never *re-serialize*;
  we only build the in-memory ModelProto and hand it
  off.

  **Hard-stop fallback if even that doesn't work**: STOP
  and flag. Do not write to a temp directory at runtime
  to enable a path-based load — that violates
  CONTRIBUTING.md §10.4 "Library takes bytes". Yuka
  picks the next move.

- The rest of Phase 0 (cooldown, revision pinning) is
  unchanged from round-02. Re-pin
  `philharmonic-connector-impl-api 0.1.0` after
  re-checking cooldown (round-02 saw `2d23h old`; should
  clear within hours of round-02's stop).

## Missing-context gating — additions

In addition to round-02's gating list:

- If `tract-onnx 0.22.x` exposes NO clean API for in-memory
  external-data (Phase 0.5 research), AND the prost-based
  ModelProto-walking fallback also doesn't work because
  tract's lower-level proto-acceptance API isn't stable:
  STOP and flag. Yuka picks between (a) writing to a
  tempfile at runtime (violates §10.4 — needs explicit
  Yuka sign-off), (b) waiting for upstream tract changes,
  (c) using a different default model.
- If bge-m3's `model.onnx` file's protobuf header doesn't
  cleanly indicate external-data via `data_location`
  field (i.e., the format is older / non-standard): STOP
  and flag.
- If the round-02-pinned bge-m3 revision SHA
  (`5617a9f61b028005a4858fdac845db406aefb181`) has been
  rebased away on HF (rare but possible): re-pin against
  current HEAD on bge-m3's `main` branch and document.
- If `prost` was added as a build-dep but causes a
  different transitive `prost` version to land in the
  workspace's runtime deps (unlikely — build-deps and
  runtime deps don't share resolution): adjust feature
  flags or pin compatibly.

All other gating from round-02 inherits.
