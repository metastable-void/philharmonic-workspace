## D1 — Substrate `MEDIUMBLOB → LONGBLOB` migration

**Crate:** `philharmonic-store-sqlx-mysql` (currently v0.1.2,
HEAD `b5d66ef` on `main`).

**Why:** `docs/design/16-embedding-datasets.md` requires
content-slot blobs that can carry deterministic-CBOR-encoded
embedding corpora. MySQL's `MEDIUMBLOB` caps at 16 MiB;
`LONGBLOB` lifts the cap to 4 GiB. The substrate currently
declares `content_bytes MEDIUMBLOB NOT NULL` in `CREATE_CONTENT`.

**What changes:**

1. In `philharmonic-store-sqlx-mysql/src/schema.rs`:
   - Update the `CREATE_CONTENT` DDL string so fresh deployments
     declare `content_bytes LONGBLOB NOT NULL`.
   - Add an idempotent ALTER migration so existing deployments
     pick up the change automatically on startup. The cleanest
     extension is to:
     - Rename the existing `INDEX_MIGRATIONS` const if it now
       covers more than indexes (e.g. `ALTER_MIGRATIONS`), or
       add a parallel `COLUMN_MIGRATIONS` const — pick whichever
       reads cleanest; document the choice with a one-line
       comment if non-obvious.
     - Run `ALTER TABLE content MODIFY COLUMN content_bytes
       LONGBLOB NOT NULL`.
     - Tolerate the case where the column is already `LONGBLOB`.
       MySQL's `MODIFY COLUMN` to the current type is a
       successful no-op, so this case usually does not raise an
       error — but match the existing
       `is_duplicate_key_name`-style pattern if any DB family in
       the LCD set (MySQL 8 / MariaDB / Aurora / TiDB) returns a
       distinguishable error class for the no-change case.
   - Existing index migrations must continue to work unchanged.
2. Bump `philharmonic-store-sqlx-mysql` from 0.1.2 → 0.1.3.
   Add a `## [0.1.3]` block to its `CHANGELOG.md` describing the
   schema change in one sentence.
3. Downstream pins of `philharmonic-store-sqlx-mysql` in
   sibling crates use caret ranges (`"0.1"`, `"0.1.0"`,
   `"0.1.1"`) — 0.1.3 satisfies all of them, no further edits
   needed.
4. Do **not** edit the workspace `[patch.crates-io]` block.

**Verification specific to D1:** `./scripts/pre-landing.sh`
auto-detects `philharmonic-store-sqlx-mysql` as modified and
runs `--ignored` (testcontainers MySQL) on it. That covers the
schema migration end-to-end against a real MySQL container.

## D2 — Optional per-job `run_timeout` override

**Crate:** `mechanics-core` (currently v0.3.1, HEAD `9e22ea3`
on `main`).

**Why:** Future embedding-dataset bulk-embed jobs will run for
minutes (one ONNX inference per item × hundreds of items),
well past the pool's default 30-second `run_timeout`. The
current pool-wide override (`MechanicsPoolConfig::run_timeout`)
is too coarse — short interactive workflows on the same pool
still want a 30 s ceiling. A per-job `Option<Duration>` lets
the lowerer attach a longer timeout to specific jobs without
affecting the rest. ROADMAP §9 specifies the shape as
"backward-compatible `Option<Duration>`".

**What changes:**

1. `MechanicsJob` (in `mechanics-core/src/internal/job.rs`):
   - Add `run_timeout: Option<Duration>` as a fourth field
     (`pub(crate)`, like the others).
   - Keep `MechanicsJob::new` signature unchanged; default to
     `None` for the new field. Add a builder-style method
     `with_run_timeout(self, timeout: Duration) -> Result<Self,
     MechanicsError>` that validates `timeout > 0` (return
     `MechanicsError::runtime_pool("job run_timeout must be >
     0")` on zero, mirroring `MechanicsPoolConfig::validate`).
   - Add a public accessor `pub fn run_timeout(&self) ->
     Option<Duration>`.
   - `Serialize` impl: serialize the field as `run_timeout`,
     skipping when `None` (manual `if let Some(d) =
     &self.run_timeout { state.serialize_field("run_timeout",
     d)?; }`). Mind the existing `serialize_struct(...,
     3)` count — make it dynamic if `run_timeout` is `Some`.
     The simplest way is a runtime-determined `len`:
     `let len = 3 + usize::from(self.run_timeout.is_some());`.
   - `Deserialize` impl: add `run_timeout: Option<Duration>`
     to `RawMechanicsJob` with `#[serde(default)]`. Construct
     via `MechanicsJob::new(...)?` then chain
     `.and_then(|j| match raw.run_timeout { Some(d) =>
     j.with_run_timeout(d), None => Ok(j) })`.
   - The existing `into_parts` returns
     `(Arc<str>, Arc<Value>, Arc<MechanicsConfig>)`. Either
     extend it to return the timeout as a fourth element and
     update all internal callers (`pool/worker.rs`,
     `pool/runtime.rs` if relevant), or leave `into_parts`
     alone and add a separate accessor used by the pool
     side. Pick whichever results in fewer mechanical
     changes; document the choice in the commit message.
2. `MechanicsPool::run` and
   `MechanicsPool::run_nonblocking_enqueue` (in
   `mechanics-core/src/internal/pool/api.rs`):
   - When the submitted `MechanicsJob` has
     `run_timeout()` returning `Some(d)`, use `d` instead of
     `self.run_timeout` for the per-call deadline calculation
     (`Self::deadline_from_timeout(...)`).
   - Otherwise fall back to `self.run_timeout` (current
     behavior).
   - There are two call sites that consume the timeout —
     update both. The intermediate
     `limited_by_run_timeout` boolean and error messages
     should still refer to the timeout that was actually
     used (`enqueue_wait` math is based on the resolved
     deadline, so it should still work without further
     changes — but verify).
3. Add tests in
   `mechanics-core/src/internal/pool/tests/` (or wherever the
   existing `run_timeout`-related tests live, e.g.
   `tests/queue.rs` already has
   `run_maps_reply_timeout_to_run_timeout`):
   - **`run_uses_per_job_timeout_when_set`** — pool default
     30 s, job override 1 s on a script that sleeps via a
     loop, expect `MechanicsError::run_timeout` within ~1 s.
   - **`run_uses_pool_default_when_job_override_absent`** —
     regression test that current behavior is preserved
     when `MechanicsJob::run_timeout()` is `None`.
   - **`run_per_job_timeout_can_exceed_pool_default`** —
     pool default 1 s, job override 5 s, on a script that
     completes in ~2 s, expect success. (This is the
     embedding use case — a long-running job on a pool
     with a short default.)
   - **`mechanics_job_serde_roundtrip_with_run_timeout`** —
     extend the existing serde test with a job that has
     `with_run_timeout(Duration::from_secs(60))` set;
     assert deserialize round-trips it.
   - **`mechanics_job_deserialize_no_run_timeout_field`** —
     legacy payload without the field deserializes with
     `run_timeout()` returning `None`.
   - Reject `with_run_timeout(Duration::ZERO)` with the
     expected error.
   - Use `MechanicsExecutionLimits` defaults from existing
     tests; the runtime-loop test pattern in
     `pool/tests/endpoint_validation.rs` is a fine model
     for the timeout tests if you need a scriptable JS body.
4. Bump `mechanics-core` from 0.3.1 → 0.3.2. Add a
   `## [0.3.2]` block to its `CHANGELOG.md` describing the
   addition (one paragraph; mention that the addition is
   serde-backward-compatible and that the `MechanicsJob::new`
   public signature is unchanged).
5. Downstream pin in `mechanics/Cargo.toml` is
   `mechanics-core = "0.3.0"` — caret-range, satisfied by
   0.3.2 — no edit needed.

**Backward compatibility:** The existing `MechanicsJob::new`
signature stays the same. The serde wire format adds an
optional field with `#[serde(default)]`. Existing payloads
without the field deserialize cleanly. No public API breaks
(additive only).

**Verification specific to D2:** `./scripts/pre-landing.sh`
auto-detects `mechanics-core` and runs lint + workspace test +
`--ignored` on it.

## D10 — WebUI code editor (CodeMirror 6)

**Sub-tree:** `philharmonic/webui` (TS / React 19, webpack
bundle). Lives in the `philharmonic` submodule. The
`philharmonic` Rust crate's version does **not** bump for WebUI
asset changes alone — the crate version reflects Rust API, not
built assets.

**Why:** Per HUMANS.md "WebUI → Code editor", the WebUI's JSON
and JS textareas are unfriendly for any non-trivial input. The
embedding-datasets WebUI (D6, post-Gate-1) needs a real
editor, and there's no reason existing JSON / JS editors
shouldn't share it. CodeMirror 6 is FLOSS (MIT), actively
maintained, modular, framework-agnostic, and has no native
deps — all of the constraints ROADMAP §9 D10 lists.

**What changes:**

1. Add CodeMirror 6 to `philharmonic/webui/package.json`. Use
   the modular `@codemirror/*` packages (not CodeMirror 5,
   not the bundled `codemirror` v0.x packages — explicit
   control over loaded language modes is required). Add as
   `dependencies` (runtime), not `devDependencies`:
   - `@codemirror/state`
   - `@codemirror/view`
   - `@codemirror/commands`
   - `@codemirror/language`
   - `@codemirror/lang-json`
   - `@codemirror/lang-javascript`
   - `codemirror` (the convenience meta-package that
     re-exports `basicSetup`)

   Pick the latest stable releases of each as of today. Run
   `npm install` (inside `philharmonic/webui/`) to update
   `package-lock.json`. Commit `package-lock.json`.
2. Create
   `philharmonic/webui/src/components/CodeEditor.tsx` — a
   thin React wrapper:
   ```tsx
   interface CodeEditorProps {
     value: string;
     onChange: (next: string) => void;
     language: "json" | "javascript";
     minHeightPx?: number;       // default ~150
     readOnly?: boolean;         // default false
     ariaLabel?: string;         // for accessibility
   }
   ```
   The component owns the `EditorView` lifecycle: mount in
   `useEffect` (constructing an `EditorState` with
   `basicSetup`, the language extension, and an
   `EditorView.updateListener.of` that calls `onChange` when
   the doc changes), destroy on unmount. When `value` changes
   from outside, dispatch a `replace` transaction only if the
   incoming string differs from the current doc (avoid
   feedback loops). When `language` changes, recreate the
   view (rare; acceptable).
3. Retrofit existing JSON / JS textareas. Each affected file
   replaces a `<textarea>` with `<CodeEditor language="..."
   value=... onChange=... />`. Keep the surrounding `<label
   className="field">` wrapper and the `<span>` label text.
   Affected sites:
   - `src/pages/Templates.tsx`:
     - line ~198: `scriptSource` (language `javascript`).
     - line ~207: `configText` (language `json`).
   - `src/pages/TemplateDetail.tsx`:
     - line ~166: `scriptSource` (language `javascript`).
     - line ~174: `configText` (language `json`).
   - `src/pages/Endpoints.tsx`:
     - line ~213: `configText` (language `json`).
   - `src/pages/EndpointDetail.tsx`:
     - line ~202: `configText` (language `json`).
   - `src/pages/Instances.tsx`:
     - line ~217: `argsText` (language `json`).
4. Add a CSS rule in `src/app.css` for the editor's outer
   wrapper. Match the existing `.field` border + radius +
   padding so it slots into existing form layouts. Default
   minimum height ~150 px; allow `max-height` with internal
   scroll. Keep the look consistent with the existing form
   (no jarring theme shift).
5. Keep the existing `parseJson(text)` helpers — the editor
   produces a `string`, and the page-level submit logic
   continues to call `JSON.parse` exactly as it does today.
   No type changes elsewhere should be needed.
6. Run `./scripts/webui-build.sh --production`. The
   committed `philharmonic/webui/dist/*` artifacts must be
   regenerated and committed in the same commit as the
   source changes.
