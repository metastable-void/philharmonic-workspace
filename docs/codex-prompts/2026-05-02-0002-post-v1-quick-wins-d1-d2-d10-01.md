## D1 â€” Substrate `MEDIUMBLOB â†’ LONGBLOB` migration

**Crate:** `philharmonic-store-sqlx-mysql` (currently v0.1.2,

**Why:** `docs/design/16-embedding-datasets.md` requires
content-slot blobs that can carry deterministic-CBOR-encoded
embedding corpora. MySQL's `MEDIUMBLOB` caps at 16 MiB;
`LONGBLOB` lifts the cap to 4 GiB. The substrate currently
declares `content_bytes MEDIUMBLOB NOT NULL` in `CREATE_CONTENT`.

**What changes:**

1. In `philharmonic-store-sqlx-mysql/src/schema.rs`:
   - Update the `CREATE_CONTENT` DDL string so fresh deployments
     declare `content_bytes LONGBLOB NOT NULL`.
   - Add an idempotent ALTER migration so existing deployments
     pick up the change automatically on startup. The cleanest
     extension is to:
     - Rename the existing `INDEX_MIGRATIONS` const if it now
       covers more than indexes (e.g. `ALTER_MIGRATIONS`), or
       add a parallel `COLUMN_MIGRATIONS` const â€” pick whichever
       reads cleanest; document the choice with a one-line
       comment if non-obvious.
     - Run `ALTER TABLE content MODIFY COLUMN content_bytes
       LONGBLOB NOT NULL`.
     - Tolerate the case where the column is already `LONGBLOB`.
       MySQL's `MODIFY COLUMN` to the current type is a
       successful no-op, so this case usually does not raise an
       error â€” but match the existing
       `is_duplicate_key_name`-style pattern if any DB family in
       the LCD set (MySQL 8 / MariaDB / Aurora / TiDB) returns a
       distinguishable error class for the no-change case.

## D2 â€” Optional per-job `run_timeout` override

**Crate:** `mechanics-core` (currently v0.3.1, HEAD `9e22ea3`
on `main`).

**Why:** Future embedding-dataset bulk-embed jobs will run for
minutes (one ONNX inference per item Ă— hundreds of items),
well past the pool's default 30-second `run_timeout`. The
current pool-wide override (`MechanicsPoolConfig::run_timeout`)
is too coarse â€” short interactive workflows on the same pool
still want a 30 s ceiling. A per-job `Option<Duration>` lets
the lowerer attach a longer timeout to specific jobs without
affecting the rest. ROADMAP Â§9 specifies the shape as
"backward-compatible `Option<Duration>`".

**Scope boundary â€” important.** D2 only extends the *Rust-side
pool wait deadline*: the `Instant`-based budget that
`MechanicsPool::run` (and `run_nonblocking_enqueue`) computes
via `deadline_from_timeout(...)` and uses for the enqueue +
reply-receive bound. The Boa runtime separately enforces
`MechanicsExecutionLimits::max_execution_time` (default 10 s)
inside `RuntimeInternal::run_source_inner` via its own
`compute_deadline` â€” that limit is **not touched by D2**. A job
with `run_timeout = Some(30 minutes)` will still be killed by
the runtime execution limit if the worker pool is configured
with a short `max_execution_time`. Provisioning the JS-side
execution limit for long-running embed scripts is **D5's
problem**, not this dispatch â€” D2 is only the prerequisite,
not the whole solution. Do not expand the scope to touch
`MechanicsExecutionLimits` in this dispatch.

**What changes:**

1. `MechanicsJob` (in `mechanics-core/src/internal/job.rs`):
   - Add `run_timeout: Option<Duration>` as a fourth field
     (`pub(crate)`, like the others).
   - Keep `MechanicsJob::new` signature unchanged; default to
     `None` for the new field. Add a builder-style method
     `with_run_timeout(self, timeout: Duration) -> Result<Self,
     MechanicsError>` that validates `timeout > 0` (return
     `MechanicsError::runtime_pool("job run_timeout must be >
     0")` on zero, mirroring `MechanicsPoolConfig::validate`).
   - Add a public accessor `pub fn run_timeout(&self) ->
     Option<Duration>`.
   - `Serialize` impl: serialize the field as `run_timeout`,
     skipping when `None` (manual `if let Some(d) =
     &self.run_timeout { state.serialize_field("run_timeout",
     d)?; }`). Mind the existing `serialize_struct(...,
     3)` count â€” make it dynamic if `run_timeout` is `Some`.
     The simplest way is a runtime-determined `len`:
     `let len = 3 + usize::from(self.run_timeout.is_some());`.
   - `Deserialize` impl: add `run_timeout: Option<Duration>`
     to `RawMechanicsJob` with `#[serde(default)]`. Construct
     via `MechanicsJob::new(...)?` then chain
     `.and_then(|j| match raw.run_timeout { Some(d) =>
     j.with_run_timeout(d), None => Ok(j) })`.
   - The existing `into_parts` returns
     `(Arc<str>, Arc<Value>, Arc<MechanicsConfig>)`. Either
     extend it to return the timeout as a fourth element and
     update all internal callers (`pool/worker.rs`,
     `pool/runtime.rs` if relevant), or leave `into_parts`
     alone and add a separate accessor used by the pool
     side. Pick whichever results in fewer mechanical
     changes; document the choice in the structured output's
     "D2 status â†’ API surface added" entry so Claude can
     mention it in the commit message.
2. `MechanicsPool::run` and
   `MechanicsPool::run_nonblocking_enqueue` (in
   `mechanics-core/src/internal/pool/api.rs`):
   - When the submitted `MechanicsJob` has
     `run_timeout()` returning `Some(d)`, use `d` instead of
     `self.run_timeout` for the per-call deadline calculation
     (`Self::deadline_from_timeout(...)`).
   - Otherwise fall back to `self.run_timeout` (current
     behavior).
   - There are two call sites that consume the timeout â€”
     update both. The intermediate
     `limited_by_run_timeout` boolean and error messages
     should still refer to the timeout that was actually
     used (`enqueue_wait` math is based on the resolved
     deadline, so it should still work without further
     changes â€” but verify).
3. Add tests in `mechanics-core/src/internal/pool/tests/`,
   following the **synthetic-worker pattern** that
   `tests/queue.rs::run_maps_reply_timeout_to_run_timeout`
   already uses: install a placeholder worker handle and never
   consume the queue, so the only thing that can fire is the

   - **`run_uses_pool_default_when_job_override_absent`** â€”

   - **`run_per_job_timeout_can_exceed_pool_default`** â€” pool

     `with_run_timeout(Duration::ZERO)` returns the expected

   For the pool tests, consult `tests/queue.rs` â€”
   `synthetic_pool`, `test_shared_with_channels`, and the
