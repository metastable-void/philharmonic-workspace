## Motivation

Per `HUMANS.md` §"Chat UI for easy testing" + ROADMAP §3.D
D13: workflows that accept `{"content": "<user_input>"}` as
input and return `{"messages": [<chat transcript in OpenAI-
style turns>]}` as output deserve an in-WebUI chat affordance,
so testing a chat-shaped workflow doesn't require hand-rolling
JSON in the existing "Execute step" form.

Backend already supports everything D13 needs:
`POST /v1/workflows/instances/{id}/execute` with `{input: {}}`
is the dual-purpose probe — on a new instance it bootstraps
the conversation by triggering the workflow's opening turn
(typically a greeting); on an existing instance it just
fetches the current transcript without adding a turn. The
workflow's JS handles both cases server-side via accumulated
context. **No backend changes required for D13.**

D13 is independent of D7/D8/D9 (Tier 2/3 connectors) and D11
(workflow authoring guide rewrite). It builds on D6
(embedding-datasets WebUI; `b581b50`) only in the sense that
the D6 dispatch settled the WebUI's pages/components/i18n
shape conventions Codex should mirror.

## Schema-detection mandate (load-bearing per HUMANS.md)

Templates that don't expose the content-in / messages-out
shape get a clean **"This workflow is not chat-compatible"**
error rather than a broken UI. Detection is **runtime
structural**, not metadata-based — there is currently no
formal I/O schema field on `TemplateDetail`.

Detection happens once on chat-tab mount:

1. Fire `executeInstance(id, {input: {}})`.
2. Inspect `response.output`:
   - If it's an object with a `messages` array, and every
     element looks like `{role, content}` (where `role` is
     a non-empty string and `content` is a string), the
     workflow is chat-compatible. Render the bubbles.
   - Otherwise: render the not-chat-compatible empty state
     with a clear message naming the actual top-level keys
     observed in `output` (so the admin can see why
     detection failed). Do **not** crash, do **not** show
     the chat input, do **not** retry.
3. If the call returns 4xx/5xx (transport or workflow error),
   surface the error as a toast and render the empty state
   with a "could not probe transcript" message + a "Retry"
   button.

The user's first real message goes through the same
`executeInstance(id, {input: {content: "..."}})` call. The
workflow's JS appends the user turn and the assistant
response to `output.messages`; the UI re-renders all bubbles
from the latest response (no client-side merging — the
server is the source of truth for the full transcript).

## Empty-content dual-purpose semantics (per ROADMAP §3.D)

Per the directive in ROADMAP §3.D ("Empty-content POST
(`{}`) is dual-purpose"):

- **On a new instance** (no prior context): the empty-
  content call bootstraps the conversation by triggering
  the workflow's opening turn — typically a greeting. The
  chat UI fires this call **automatically on first chat-
  tab mount** so admins don't have to send a dummy message
  to see the assistant's opening.
- **On an existing instance**: the empty-content call just
  fetches the current transcript without adding a turn.
  Useful for tab re-opens, page refresh, sharing a chat
  URL.

The workflow's script handles both cases server-side via
accumulated context. **The UI does NOT distinguish between
the two cases** — it always fires the empty-content call
on first mount, then renders whatever transcript comes
back. Differentiation is the workflow's responsibility,
not the UI's.

The implication: the chat tab is **always** safe to
navigate to from a fresh URL; admins land on a populated
chat (whether bootstrapped greeting or existing
transcript) without needing to send a dummy turn.

## Deliverable A — Types & API client (`api/client.ts`)

Add chat-decoded types alongside the existing
`ExecuteInstance{Request,Response}` types. The existing
wire types are unchanged — these are **decoded views** of
`response.output` for the chat shape.

```ts
export type ChatRole = "user" | "assistant" | "system" | string;

export interface ChatMessage {
  role: ChatRole;
  content: string;
  /** Workflows may attach extra fields per OpenAI conventions
   *  (`name`, `tool_calls`, etc.); preserve them as opaque
   *  passthrough so the UI doesn't drop information when it
   *  re-renders the transcript on the next turn. */
  [extra: string]: JsonValue | string;
}

export interface ChatTranscript {
  messages: ChatMessage[];
}

export type ChatDetection =
  | { kind: "chat"; transcript: ChatTranscript }
  | { kind: "not-chat"; observedKeys: string[] };

/** Inspect a workflow execute response and decide whether
 *  it conforms to the chat-style I/O contract. */
export function parseChatOutput(output: JsonValue): ChatDetection;
```

`parseChatOutput` rules:

- `output` must be a non-null object.
- `output.messages` must be an array.
- Every element of `messages` must be a non-null object with
  `role: string` (non-empty) and `content: string` (string,
  may be empty).
- If any element fails the shape check, the whole detection
  is `not-chat`.
- The `observedKeys` array on `not-chat` returns the
  top-level keys of `output` (for the empty-state error
  message), or `[]` if `output` was null / not an object.

Add fetch helpers if missing (inspect `client.ts` first;
don't duplicate existing ones):

- `executeInstance(id: string, req: ExecuteInstanceRequest):
  Promise<ExecuteInstanceResponse>` — likely already exists
  inline in `InstanceDetail.tsx` as an `apiCall<...>` call;
  extract to `client.ts` if reuse is non-trivial, or inline
  it consistently in the chat tab if extraction would force
  a wider refactor. Match existing style.
- `createInstance(req: CreateInstanceRequest):
  Promise<CreateInstanceResponse>` — likely missing
  (existing `Instances.tsx` may or may not have one). Add
  it if missing.

JSDoc all new exports.

## Deliverable D — Chat tab UI

Lives inside `InstanceDetail.tsx`'s tab content area. The
content is conditionally rendered based on
`activeTab === "chat"`.

State:

```tsx
const [chatDetection, setChatDetection] =
  useState<ChatDetection | null>(null);
const [chatProbeError, setChatProbeError] =
  useState<string | null>(null);
const [chatInflight, setChatInflight] = useState(false);
const [draft, setDraft] = useState("");
```

Mount-effect (fires when chat tab first becomes active —
gate on a `chatProbed: boolean` flag so it fires exactly
once per InstanceDetail mount, not on every tab switch):

1. `setChatInflight(true)`.
2. `executeInstance(id, { input: {} })`.
3. On success: `setChatDetection(parseChatOutput(
   response.output))`.
4. On failure: `setChatProbeError(messageFrom(caught,
   t.common.requestFailed))`.
5. `setChatInflight(false)`.

Render branches:

### Branch 1: detection pending
```
[skeleton bubbles or "loading transcript..." spinner]
```
Use `t.chat.probing` or similar for the loading text.

### Branch 2: detection = `not-chat`
```
[empty state with message:
  "This workflow is not chat-compatible.
   Expected `{messages: [{role, content}, ...]}` in output;
   got top-level keys: <observedKeys>."
 plus a link to the existing "Execute step" tab/form so
 admins can fall back to the raw JSON path.]
```
Use `t.chat.notChatCompatible(observedKeys)` (i18n function
that interpolates the keys list).

### Branch 3: detection = `chat`
- **Transcript pane**: vertically scrollable, alternating
  bubble alignment (user = right, assistant / system =
  left). Bubbles show `content` as plain text — no markdown
  rendering for D13 (out of scope; flag in residual risks
  if Yuka would benefit from markdown).
  - Roles other than `user` / `assistant` / `system` get a
    neutral / "other" bubble style + a small role label.
  - Use existing color palette / spacing — no new design
    tokens.
- **Autoscroll**: on transcript change, scroll the bubble
  container to the bottom. Save scroll position to
  localStorage on unmount (`chat:scroll:<instanceId>`); on
  remount, restore. **Don't autoscroll** if the user has
  scrolled up from the bottom (preserve their reading
  position when new turns arrive); detect "at bottom" via
  scroll-event delta. Standard chat-UI behavior.
- **In-flight indicator**: while a send is in flight, show
  a typing/spinner placeholder bubble at the bottom of the
  transcript (assistant-aligned) until the response
  arrives. Disable the input + send button.
- **Composer**: textarea + send button at the bottom.
  - **Send-on-Enter**: Enter sends; Shift+Enter inserts

    1. Trim the draft.
    2. `setChatInflight(true)`.

- **Retry button** (when probe failed and `chatProbeError`

CSS: prefer existing utility classes (`stack`, `panel`,

## Deliverable E — localStorage helpers

Add a small helpers module — either inline in
`InstanceDetail.tsx` / `TemplateDetail.tsx` if usage stays
small, or extract to `philharmonic/webui/src/util/
chatStorage.ts` if both pages duplicate. Codex's call.

Two keys, namespaced under `chat:`:

- `chat:last-instance:<templateId>` — instance UUID. Set
  whenever the chat tab successfully runs (probe or send),
  on InstanceDetail. Read on TemplateDetail to decide
  whether to show the "Continue last chat" option. Cleared
  if the instance returns 404 on probe (the instance was
  retired or deleted out from under us).
- `chat:scroll:<instanceId>` — scroll position (number,
  pixels). Set on chat-tab unmount. Read on chat-tab mount
  after the initial render.

**Quota / corruption tolerance**: localStorage operations
must wrap in `try/catch` — quota-exceeded and disabled-
storage errors should be silently swallowed (the chat UI
still works without persistence; the cache is purely
optional).

**Per-template-id key**: this means a one-template tenant's
last-instance shortcut works as expected. It does NOT
provide cross-template chat history.

**No transcript persistence**: per HUMANS.md, "No
persistence is required immediately beyond simple
localStorage usages." The transcript itself comes from
the workflow's accumulated context server-side; do **not**
cache it client-side. (Caching invites stale-render bugs
when two tabs hit the same instance.)
