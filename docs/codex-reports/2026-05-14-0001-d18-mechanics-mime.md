# D18 mechanics MIME module notes

**Date:** 2026-05-14
**Prompt:** docs/codex-prompts/2026-05-14-0004-d18-mechanics-module-surface-03.md

Round 03 landed `mechanics:mime` with a smaller dependency shape
than the prompt's suggested `mailparse` + `mail-builder` pair.
The new `mime` feature activates the already-present
`data-encoding` crate as the Base64 backing dependency; the rest
of the MIME formatter/parser is in-module code.

The reason is twofold. First, the local registry cache was
read-only during the round, so `cargo info quoted_printable`
could discover metadata but could not install a fresh crate into
the configured cache. Second, the JS API is intentionally a
format-only structured object surface rather than a full email
address/mailbox model, so `mail-builder`'s email-message builder
shape would still require adaptation for arbitrary user-supplied
headers and message trees.

Residual RFC scope is deliberately narrow. Header encoded-word
handling supports UTF-8 `B` and `Q` encoded words; legacy
charsets are rejected. Header folding is whitespace-based and
does not attempt optimal RFC 2047 encoded-word splitting for very
long non-ASCII headers. Multipart parsing ignores preamble and
epilogue content, recursively parses parts, and trims the
delimiter newline that separates each body part from the next
boundary. These are acceptable for the initial Mechanics module
surface, but they are worth revisiting before treating
`mechanics:mime` as a broad email interoperability layer.
