# Decision: fix `Sha256` CBOR encoding in `philharmonic-types 0.3.5`

**Date:** 2026-04-22
**Decided by:** Yuka
**Decision:** Option (A) from
  `docs/notes-to-humans/2026-04-22-0009-sha256-cbor-encoding-blocker.md`.

## What this means

`philharmonic-types 0.3.5` will land a human-readable-aware
serde impl for `Sha256`: hex text string for
`is_human_readable() == true` (JSON, YAML, TOML — no behavior
change for existing consumers), 32-byte byte string for
`is_human_readable() == false` (CBOR — the COSE-right shape).

This matches what the `uuid` crate already does for `Uuid`.

## Rollout

1. Patch `philharmonic-types/src/sha256.rs`; add JSON + CBOR
   serde tests covering both shapes.
2. Bump `philharmonic-types` to `0.3.5`, update its
   `CHANGELOG.md`, publish via `./scripts/publish-crate.sh`.
3. Bump the `philharmonic-types` pin to `0.3.5` (or leave caret
   `"0.3"` where that's already in place — audit per-crate)
   in the downstream crates, starting with the ones that ship
   in v1:
   - `philharmonic-policy`
   - `philharmonic-store`
   - `philharmonic-store-sqlx-mysql`
   - `philharmonic-connector-common`
   - `philharmonic-workflow`
4. Revise the Wave A proposal r3 → r4. The encoding claim
   ("`Sha256` serializes as a 32-byte byte string") is now true
   (after 0.3.5 lands). Add a short note tying the encoding to
   `philharmonic-types >= 0.3.5`.
5. Generate the pycose reference vectors against the fixed
   encoding, commit them, write + archive the Codex prompt,
   dispatch Codex.

## Why this is not a Gate-1 trigger

The serde impl change is not a crypto primitive. `Sha256::of`
still computes SHA-256 via `sha2`; the change is purely about
wire-format encoding of an already-computed hash. Housekeeping,
not Gate-1-sensitive.

The Wave A proposal itself is Gate-1 approved; this rollout is
about aligning the upstream types crate with what the approval
described.
