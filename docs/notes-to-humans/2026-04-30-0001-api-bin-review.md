# Phase 9 task 4c — `philharmonic-api` bin review

**Date**: 2026-04-30
**Author**: Claude Code

## What landed

The `philharmonic-api` binary wires the full API stack into a
runnable server:

- **Database**: `SinglePool::connect(url)` → `SqlStore` (no
  direct `sqlx` dependency in the meta-crate — the store-sqlx
  crate wraps it).
- **Signing key**: Ed25519 seed loaded from file →
  `ApiSigningKey::from_seed(seed, kid)`.
- **Verifying key registry**: Multiple verifying keys loaded
  from config entries (kid + public key path + issuer +
  validity window).
- **SCK**: Optional substrate confidentiality key from file →
  `Sck::from_bytes`.
- **Connector router**: Embedded via `extra_routes` — dispatch
  table from config, nested at `/connector/`.
- **Scope resolver**: `HeaderBasedScopeResolver` reads
  `X-Tenant-Id` header → `RequestScope::Tenant`. No header →
  `Operator`. This is a working placeholder for deployment-
  specific resolution.
- **Stubs**: `StubExecutor` and `StubLowerer` for the workflow
  engine (real impls come in the e2e integration task).
- **Rate limiting**: Configurable per-family overrides, defaults
  from the library crate's `RateLimitConfig::default()`.
- **SIGHUP reload**: Rebuilds verifying key registry + connector
  dispatch + rate limit config. DB pool and signing key are
  long-lived (not reloaded).
- **Dynamic router**: `Arc<RwLock<Router>>` with `tower::oneshot`
  delegation, allowing hot-swap of the entire API + connector
  router on SIGHUP.

## Design decisions

1. **No direct `sqlx` dep**: The initial Codex output depended
   on `sqlx` directly for `MySqlPool::connect`. Claude added
   `SinglePool::connect(url)` to `philharmonic-store-sqlx-mysql`
   instead + made `SinglePool` `Clone`. This keeps the meta-
   crate's dep surface smaller and avoids leaking backend
   implementation details.

2. **`HeaderBasedScopeResolver`**: A placeholder — real
   deployments will resolve scope from subdomain, JWT, or
   similar. The `X-Tenant-Id` header approach is enough for
   testcontainers e2e without coupling to a specific auth
   strategy.

3. **`synthetic_public_uuid`**: The scope resolver needs an
   `Identity { internal, public }` pair. Since the bin doesn't
   have a real identity-resolution flow, it generates a
   synthetic public UUID from the internal one by setting
   UUID version/variant bits. This is a test/dev convenience,
   not a production pattern.

## What's NOT included

- WebUI embedding (no React source tree yet).
- Real `StepExecutor` / `ConfigLowerer` (stubs for now).
- Real scope resolution (header-based placeholder).
- Tests (e2e testcontainers task).
